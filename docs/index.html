§ HOW IT WORKS

Plan first. Preview before you build. Run it on your own box.

Taskloom doesn't dump a wall of code on you. It listens, thinks, then proposes — so you ship deliberate, reviewable internal tools instead of disposable demos.

01

Describe what you need

Type a paragraph. "A CRM for ten account managers." "An agent that drafts replies to support email at 9am." Taskloom asks a couple of clarifying questions, then commits to a brief.

02

Plan & preview

Taskloom turns the brief into a step-by-step plan and a clickable preview you can poke at. Disagree with a step? Edit it. The preview re-renders before any code is committed.

03

Run & monitor

Publish to your workspace. Watch every run stream in real time over SSE. Schedule with cron. Trigger via webhook. All evidence is logged so you can audit what shipped and why.

§ FEATURES

Everything you need to stand up internal tooling. Nothing you don't.

Built for operators and engineers who want one box that hosts the planner, the preview, the runtime, the secrets, and the audit log — and nothing leaves it.

Brief & plan engine

Every build starts with a structured brief and a numbered plan. Iterate on intent before code, not after a thousand-line PR.

Multi-provider routing

Anthropic, OpenAI, MiniMax, Ollama. Pick a model per agent. Failover or pin. Your keys, your bills, your traffic.

Persistent jobs & cron

Schedule any agent like a cron job. Long-running jobs survive process restarts and resume cleanly.

Streaming run console

Server-sent events stream every tool call, every model token, every error to a live console you can scroll, filter, and replay.

Encrypted secrets vault

API keys and credentials are sealed with AES-256-GCM and only unlocked at runtime by the agent that owns them.

Webhooks in & out

Trigger any agent from a webhook. Have agents post results to Slack, GitHub, Linear, or your own endpoints. JSON in, JSON out.

Workspace activation

See which agents are actually used, which are idle, and which produced evidence this week. Activation, not just deploy.

Audit & activity feed

Every run, every edit, every secret access is recorded. Filter by user, agent, or outcome. Export for your auditors.

Sandboxed runtime

Generated code runs in a Docker-isolated sandbox with explicit network and filesystem allowlists. No surprise side effects.

§ SELF-HOST

Your data, your models, your box.

Clone the repo, install dependencies, point it at a model. Taskloom runs anywhere Node runs — your laptop, a Hetzner VM, or a corporate VPC.

  • MIT licensed — fork it, ship it inside your company, build a product on top.
  • Single repo, one process per service — an API on :8484 and a web app on :7341, no orchestrator required.
  • Bring your own model keys — Anthropic, OpenAI, MiniMax, or a local Ollama. No middle-tier proxy.
  • JSON for local dev, SQLite or Postgres for deploys — choose the store that matches your topology.
  • Encrypted secrets vault — AES-256-GCM at rest, scoped per workspace.
  • No telemetry, ever — nothing phones home. There is no analytics endpoint to disable; there isn't one.
  • Docker-friendly — runs in a container, runs the sandboxed runtime in a container, runs you in a container if that's your thing.

§ FAQ

Questions, plainly answered.

Is this really self-hosted?
Yes. Clone the repo, run npm run dev, and the entire stack — API, web, planner, runtime, secrets vault, audit log — lives on your machine. There is no SaaS dependency to opt out of, because there isn't one.
What models do I need?
Any one of Anthropic, OpenAI, MiniMax, or a local Ollama instance is enough to get going. You configure a key per provider, then pick which model handles each app or agent workflow.
How is this different from v0, Bolt, or Lovable?
Two ways. First: Taskloom plans before it codes — you get a brief and a numbered plan you can edit, instead of a wall of generated code. Second: Taskloom is yours. Self-hosted, MIT, no telemetry, no rate limits other than the ones your provider charges you for.
Is there a hosted version?
No. Taskloom ships as source you run yourself, and the project does not plan to operate a managed Taskloom service.
Can I run it without Docker?
Yes. Node 20+ and a database connection string is all the API needs. Docker is only required if you want the sandboxed code-execution runtime — and even that is opt-in per agent.
What database does it use?
File-backed JSON is the default contributor store. Use SQLite for a single-node deployment, or switch to Postgres with one connection string when you need horizontal app writers. Migrations run on boot.
